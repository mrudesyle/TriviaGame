How can you support the SELECT clause of a query by using a nonclustered index that is already used for the WHERE clause?

 You could use SELECT *.
 You could modify the index that is already used to include the columns from the select list that are not part of the key.
 You could add column aliases.
 There is no way to support the SELECT clause with indexes

Where does SQL Server sort the data, if a sort is needed?

 In the current database
 In the master database
 In the msdb database
 SQL Server sorts data in memory, or spills the data to tempdb if it does not fit in memory

Why are window functions allowed only in the SELECT and ORDER BY clauses of a query?

 Because they are supposed to operate on the underlying query’s result, which is achieved when logical query processing gets to the SELECT phase.
 Because Microsoft didn’t have time to implement them in other clauses.
 Because you never need to filter or group data based on the result of window functions.
 Because in the other clauses, the functions are considered door functions (also known as backdoor functions)

How can you turn off nested triggers on a SQL Server Instance by using T-SQL?

 Use the sp_configure stored procedure follwed by 'nested triggers' and 'OFF'.
 Use the sp_configure stored procedure followed by 'nested triggers' and 0.
 Use the sp_configure stored procedure followed by 'nested triggers' and 'OFF', followed by the RECONFIGURE statement.
 Use the sp_configure stored procedure followed by 'nested triggers' and 0, followed by the RECONFIGURE statement.

Can you have indexes on table variables?

 No
 Yes, by running the CREATE INDEX command
 Yes, indirectly by defining primary key and unique constraints
 Yes, by defining foreign keys
